# 事故响应 Runbook（P0/P1/P2）

## 定义
- **P0**：全站不可用/核心交易不可用/重大安全事件
- **P1**：核心功能严重退化、有明确客户影响
- **P2**：局部功能受影响，可绕过

## 1. 事故处理的目标
1. **先止血**（恢复服务/降低影响）
2. **再定位**（找到最可能根因）
3. **后复盘**（形成可行动项）

## 2. On-call 处理流程（简版）
1. 建立事件频道（Slack/Teams）+ 记录人
2. 打开 Status Page（如有）并声明影响范围
3. 优先检查最近变更（发布、配置、依赖升级）
4. 若可快速回滚 → 先回滚
5. 收集证据：时间线、错误率、关键日志、变更单号
6. 每 15–30 分钟更新一次状态

## 3. 什么时候升级
- 影响扩大、或者 30 分钟内无法止血 → 升级 P0/P1
- 涉及数据一致性/资金/隐私 → 立即拉合规与安全

## 4. 复盘要求（最少字段）
- 影响时间窗
- 根因（直接/间接）
- 触发条件（为什么这次发生）
- 检测与响应（为何没提前发现）
- 行动项（owner + 截止时间）
